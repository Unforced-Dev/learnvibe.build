02 The Frontier: OpenClaw

What's happening right now

An open-source autonomous AI agent called OpenClaw (originally Clawdbot, then Moltbot) has gone viral. It's built on MCP — the same protocol we learned about last week — and has over 150,000+ GitHub stars.

This is an AI that can run tasks, use tools, and act autonomously. It's a glimpse of where things are heading — and a reminder of why we need to understand what we're building with.

Why this matters

OpenClaw shows how quickly the frontier is moving. The tools we're learning aren't just for making websites — they're the foundation for autonomous systems that can take real action in the world.

The MCP connection

Remember MCPs from Week 2? OpenClaw is built on that same protocol. The "connectors" that let Claude access your files and services are the same infrastructure powering autonomous agents.

What to pay attention to

This is moving fast. Things that seem experimental today become standard tomorrow. Stay curious, but also stay grounded. Understanding the foundations matters more than chasing every new tool.

03 The Paradox of Slowing Down

"The faster the tools get, the slower we need to think."

This might be the most important lesson of this week.

Claude Code can generate hundreds of lines of code in seconds. It can refactor entire projects. It can spin up databases and deploy applications. This power is intoxicating — and dangerous if you're not careful.

Speed creates urgency

When you can build fast, there's pressure to build more. To add features. To "just try one more thing." Before you know it, you've created complexity you don't understand.

Complexity compounds

Each feature you add creates surface area for bugs. Each integration creates dependencies. The AI doesn't feel this complexity — but you will when things break.

Understanding matters

You don't need to understand every line of code. But you need to understand what your system does, what it touches, and what could go wrong. If you can't explain it, slow down.

Practices for staying grounded

→ Take breaks. Step away regularly. Come back with fresh eyes.
→ Write first, build second. Articulate what you want before asking Claude to build it.
→ Review what was built. Ask Claude to explain what it created. Walk through it together.
→ Trust your felt sense. If something feels off, pause. Your intuition matters.

04 Git and GitHub

Before we dive into Claude Code, we need to understand Git and GitHub. These are fundamental to how software is built — and how Claude Code works under the hood.

What is Git?

Git is version control. It tracks every change to your code, so you can:

  • • Go back in time — undo mistakes, restore previous versions
  • • Branch — try experiments without breaking the main code
  • • Merge — bring changes together safely
  • • Collaborate — multiple people working on the same project

What is GitHub?

GitHub is where Git repositories live online. Think of it as:

  • • Cloud storage for code — backup and sync your projects
  • • Collaboration hub — share code, get contributions
  • • Free hosting — GitHub Pages can serve your websites
  • • The world's code library — millions of open source projects

Key Concepts

Repository (Repo)

A folder that Git tracks. Contains your code plus the history of every change.

Commit

A snapshot of your code at a moment in time. Like a save point in a video game. Each commit has a message describing what changed.

Branch

A parallel version of your code. The main branch is usually called "main." You can create branches to try things, then merge them back.

Worktree

A way to have multiple branches checked out at once in different folders. Claude Code uses worktrees by default when integrated with Claude Desktop — this lets you work on code while keeping the original safe.

Important: What NOT to commit

  • • API keys and secrets — never put these in code
  • • .env files — contain environment variables and secrets
  • • Personal data — don't commit user info or private data
  • • Large files — images, videos, databases

Use a .gitignore file to tell Git what to exclude.

05 Claude Code Deep Dive

Claude Code is Claude in your terminal — a full agentic coding assistant that can read, write, and execute code on your machine.

What makes it different

Full file system access

Claude Code can read and write files anywhere you give it permission. It sees your whole project, not just what you paste in.

Runs commands

It can run shell commands — install packages, run tests, start servers, deploy to production. Real power, real responsibility.

Understands context

Claude Code reads your codebase to understand how things fit together. It can navigate between files, understand dependencies, and make coherent changes across your project.

Two ways to access

Terminal: Run claude in any folder. Full control, cross-platform.
Claude Desktop: Integrated as a feature. Uses Git worktrees for safety.

Getting started

# Install Claude Code (requires Node.js)
npm install -g @anthropic-ai/claude-code

# Navigate to your project
cd my-project

# Start Claude Code
claude

Claude will read your project and be ready to help.

Useful commands

/help See all available commands
/clear Clear conversation history
/compact Summarize conversation to save context
/cost See how much this session has cost

The skill is the same

Claude Code is more powerful, but the core skill is identical: articulate what you want clearly. Know what you're building. Stay grounded.

06 Building Something Real

The path from idea to working software. This is where everything comes together.

The workflow

01

Idea

Start with your felt sense. What are you trying to create? Who is it for? What problem does it solve? This is thinking with self.

02

PRD (Product Requirements Document)

Write down what you're building. Not how — what. Features, user flows, success criteria. Claude can help you create this.

"Help me write a PRD for [your idea]. I want to build [description]. The user should be able to [key features]."

03

Technical Spec

Now we figure out how. What technologies? What architecture? What files and components? Claude can help translate the PRD into a spec.

"Based on this PRD, what's the simplest way to build this? I'm new to coding — keep it simple."

04

Research

Before building, understand what exists. What libraries? What patterns? What can we learn from similar projects? Claude can search and summarize.

05

Build

Now we build. Start small. Get something working. Then iterate. Don't try to build everything at once.

"Let's start with the simplest version. Just [core feature]. We can add more later."

06

Test & Iterate

Does it work? Does it feel right? Get feedback. Make changes. This is where thinking with others comes in — show people, learn, improve.

Start smaller than you think

The most common mistake is trying to build too much. Start with the tiniest possible version that does something useful. You can always add more.

07 Care and Permissions

Claude Code has real power. It can modify files, run commands, and make changes to your system. With great power comes the need for care.

Permission prompts matter

Claude Code asks for permission before taking actions. Read these prompts carefully. Understand what you're approving before you say yes.

  • • File writes — what file is being modified?
  • • Command execution — what will this command do?
  • • Network access — is it reaching out to the internet?

Best practices

Use Git

Commit your work before making big changes. If something goes wrong, you can always go back. Git is your safety net.

Review changes

After Claude makes changes, look at what it did. Ask it to explain. Use git diff to see exactly what changed.

Work in a sandbox first

When trying something risky, do it in a separate folder or branch. Don't experiment on code you can't afford to break.

Keep secrets out

API keys, passwords, personal data — keep these in environment variables or separate files that aren't tracked by Git.
